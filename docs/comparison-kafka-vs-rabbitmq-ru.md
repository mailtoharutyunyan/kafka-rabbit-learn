# Kafka vs RabbitMQ — Сравнение и выбор

## Философия

| | Kafka | RabbitMQ |
|---|---|---|
| **Модель** | Распределённый лог (append-only) | Брокер очередей (AMQP) |
| **Сообщения** | Хранятся, пока не истечёт retention | Удаляются после подтверждения |
| **Потребители** | Pull (consumer запрашивает данные) | Push (broker отправляет потребителю) |
| **Порядок** | Гарантирован в рамках партиции | Гарантирован в рамках одной очереди |
| **Прочитано = удалено?** | Нет, сообщения остаются | Да (после ack) |

## Архитектурные отличия

```
╔═══════════════════════════════════════════════════════════════════╗
║                          KAFKA                                    ║
║                                                                   ║
║   Producer ──→ Topic ──→ Partition (лог на диске)                ║
║                                    │                              ║
║                          Consumer Group A (offset 5)              ║
║                          Consumer Group B (offset 3)              ║
║                                                                   ║
║   → Сообщения ОСТАЮТСЯ в логе (retention: 7 дней, ∞, compact)   ║
║   → Каждая группа ведёт свой offset                              ║
║   → Можно перемотать назад                                       ║
╚═══════════════════════════════════════════════════════════════════╝

╔═══════════════════════════════════════════════════════════════════╗
║                        RABBITMQ                                   ║
║                                                                   ║
║   Producer ──→ Exchange ──routing──→ Queue ──→ Consumer           ║
║                                        │                          ║
║                              (ack) → удалить                     ║
║                              (nack) → вернуть / DLQ              ║
║                                                                   ║
║   → Сообщение УДАЛЯЕТСЯ после обработки                          ║
║   → Гибкая маршрутизация через exchanges/bindings                ║
║   → Нельзя «перемотать» очередь                                  ║
╚═══════════════════════════════════════════════════════════════════╝
```

## Сравнительная таблица

| Критерий | Kafka | RabbitMQ |
|----------|-------|----------|
| **Throughput** | Очень высокий (100K–1M+ msg/sec) | Средний (10K–50K msg/sec) |
| **Latency** | Выше (батчинг, replication) | Ниже (push-модель) |
| **Хранение** | На диске, настраиваемый retention | В памяти + диск (пока не ack) |
| **Масштабирование** | Горизонтальное (партиции) | Вертикальное + кластеризация |
| **Маршрутизация** | Простая (по топику/ключу) | Сложная (exchange types, bindings) |
| **Replay** | Да (перемотка offset) | Нет (сообщение удалено) |
| **Порядок** | В рамках партиции | В рамках очереди |
| **Протокол** | Свой бинарный | AMQP 0-9-1 (стандарт) |
| **Стриминг** | Kafka Streams, ksqlDB | Нет встроенного |
| **Приоритеты** | Нет | Есть (priority queues) |
| **TTL** | Через retention (весь топик) | На уровне сообщения/очереди |
| **Транзакции** | Есть (exactly-once) | Channel transactions + confirms |
| **RPC** | Возможно, но не идиоматично | Встроенный паттерн |
| **Отложенные сообщения** | Нет нативно | DLX + TTL |
| **Management UI** | Kafka UI / Confluent Control Center | Встроенный (порт 15672) |
| **Язык** | Java/Scala | Erlang |

## Когда выбрать Kafka

```
✅ Kafka — лучший выбор, когда:

• Нужен высокий throughput (логи, метрики, IoT, clickstream)
• Event sourcing — нужно хранить историю событий
• Stream processing — обработка в реальном времени
• Нужна возможность replay (перечитать данные)
• Несколько consumer groups читают одни данные
• Change Data Capture (CDC) из БД
• Log aggregation из множества сервисов
• Данные должны храниться долго (дни, месяцы)
• Порядок критичен для ключа (user_id, order_id)
```

## Когда выбрать RabbitMQ

```
✅ RabbitMQ — лучший выбор, когда:

• Сложная маршрутизация (topic patterns, headers matching)
• Request/Reply (RPC) между сервисами
• Приоритеты задач (priority queues)
• Отложенная обработка (delayed messages via DLX+TTL)
• Нужен стандартный протокол (AMQP, STOMP, MQTT)
• Низкая latency важнее throughput
• Task queues (аналог Celery)
• Лёгкий setup, меньше инфраструктуры
• Routing logic часто меняется (bindings без перезапуска)
• Сообщение обрабатывается ровно одним потребителем
```

## Типичные use-cases

### E-commerce

| Задача | Рекомендация |
|--------|-------------|
| Поток заказов (event sourcing) | **Kafka** — хранение истории, replay |
| Уведомления (email, push) | **RabbitMQ** — routing по типу |
| Обработка платежей | **Kafka** — exactly-once, порядок |
| Генерация PDF/отчётов | **RabbitMQ** — work queue с приоритетом |
| Analytics pipeline | **Kafka** — высокий throughput |
| Межсервисный RPC | **RabbitMQ** — встроенный паттерн |

### Микросервисы

| Задача | Рекомендация |
|--------|-------------|
| Event-driven архитектура | **Kafka** — события хранятся |
| Saga / orchestration | **RabbitMQ** — flexible routing |
| CQRS (проецирование) | **Kafka** — replay для перестроения view |
| Background jobs | **RabbitMQ** — work queues |
| Audit log | **Kafka** — неудаляемый лог |
| API Gateway → services | **RabbitMQ** — low latency RPC |

## Можно ли использовать оба?

**Да!** Это распространённая практика:

```
┌──────────────────────────────────────────────────────┐
│                  Hybrid Architecture                   │
│                                                       │
│  [Сервис A] ──event──→ KAFKA ──→ [Analytics]         │
│       │                   │                           │
│       │                   └──→ [Event Store]          │
│       │                                               │
│       └──command──→ RABBITMQ ──→ [Сервис B]          │
│                         │                             │
│                         └──→ [Notification Service]   │
└──────────────────────────────────────────────────────┘
```

- **Kafka** для event stream (что произошло)
- **RabbitMQ** для commands (что нужно сделать)
